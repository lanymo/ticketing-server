# 대용량 트래픽 처리 티켓팅 서비스 기획서

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | 고성능 티켓팅 시스템 |
| 목표 | 동시 대규모 요청을 안정적으로 처리하는 티켓팅 플랫폼 구축 |
| 핵심 문제 | 특정 시점에 수만~수십만 명이 동시 접속하여 티켓 구매를 시도하는 상황 처리 |

---

## 2. 기술 스택

| 레이어 | 기술 |
|--------|------|
| Backend | Spring Boot 3.x, Java 21 |
| Database | MySQL (메인 DB), Redis (캐시/분산락/대기열) |
| Message Queue | Apache Kafka (이벤트 스트리밍), RabbitMQ (결제/알림 작업 큐) |
| Cache | Redis |
| API Gateway | Spring Cloud Gateway |
| 서비스 등록/발견 | Eureka (Spring Cloud Netflix) |
| 컨테이너 | Docker, Docker Compose |
| 모니터링 | Prometheus, Grafana |

## 4. 핵심 동작 흐름

### 4-1. 대기열 처리 흐름 (Queue Flow)

```
사용자 접속
    │
    ▼
Redis Sorted Set에 진입 요청 등록
(score = 타임스탬프, member = userId)
    │
    ├─── 대기 순번 조회 응답 → 클라이언트 폴링
    │
    ▼
Scheduler (1초 주기): 앞 N명씩 입장 토큰 발급
    │
    ▼
입장 토큰 발급 → Redis에 TTL 설정 (예: 5분)
    │
    ▼
예매 페이지 접근 허용 (토큰 검증 통과 시)
```

### 4-2. 티켓 예매 흐름 (Reservation Flow)

```
[클라이언트] 좌석 선택 + 예매 요청
    │
    ▼
[API Gateway] 입장 토큰 검증 + Rate Limit 체크
    │
    ▼
[Ticket Service]
    ├── Redis 분산 락 획득 (Redisson, 좌석 단위 락)
    ├── Redis 재고 확인 (DECR 원자적 연산)
    ├── 재고 있음 → 임시 예약 정보 Redis에 저장 (TTL: 10분)
    └── 재고 없음 → 409 Conflict 응답
    │
    ▼
[Kafka Producer] ticket-reserved 토픽으로 이벤트 발행
    │
    ▼
[Kafka Consumer → Payment Service]
    ├── RabbitMQ payment.queue에 결제 작업 전송
    │
    ▼
[Payment Service]
    ├── 외부 PG사 API 호출
    ├── 결제 성공 → MySQL에 최종 예약 저장
    └── 결제 실패 → Redis 재고 롤백 + 취소 이벤트 발행
    │
    ▼
[Notification Service]
    └── 이메일/SMS 발송 (RabbitMQ notification.queue)
```

## 5. 마이크로서비스 구성

### 5-1. User Service
- 회원가입 / 로그인 (JWT 발급)
- 세션 토큰을 Redis에 저장
- 역할: 인증/인가 담당

### 5-2. Event Service
- 공연 정보 CRUD
- 좌석 배치도 관리
- 이벤트 오픈 스케줄 관리
- 공연 정보 Redis 캐싱 (자주 조회되는 데이터)

### 5-3. Ticket Service (핵심 서비스)
- 대기열 진입/순번 조회
- 입장 토큰 발급
- 좌석 선택 및 임시 예약
- 분산 락 기반 동시성 제어
- Redis 재고 관리

### 5-4. Payment Service
- RabbitMQ consumer로 결제 요청 처리
- 외부 PG 연동 (토스, 카카오페이 등)
- 결제 성공/실패 처리 및 이벤트 발행
- 환불 처리

### 5-5. Notification Service
- RabbitMQ consumer로 알림 처리
- 이메일 발송 (예매 확인, 취소, 입장 토큰 발급)
- SMS 발송
- 푸시 알림

## 6. 동시성 제어 전략

### 6-1. Redis 분산 락 (Redisson)
- 같은 좌석에 대한 동시 예매 요청 방지
- 락 단위: `lock:seat:{eventId}:{seatId}`
- TTL: 3초 (데드락 방지)
- 획득 실패 시 즉시 실패 응답 (No-Wait 방식)

### 6-2. Redis 원자적 재고 관리
- `DECR` 명령어로 재고 원자적 감소
- 재고 0 이하 시 예약 거부
- 결제 실패/취소 시 `INCR`로 재고 복구

### 6-3. 대기열 (Redis Sorted Set)
- Key: `queue:{eventId}`
- Score: 진입 타임스탬프 (선착순 보장)
- Scheduler가 주기적으로 앞 N명씩 입장 처리
- 입장 토큰: `token:{userId}:{eventId}` → Redis TTL 설정

### 6-4. Rate Limiting (API Gateway)
- IP 단위 요청 제한: 초당 10건
- 사용자 단위 요청 제한: 초당 5건
- Redis 기반 슬라이딩 윈도우 알고리즘

## 8. 장애 대응 전략

| 장애 상황 | 대응 방법 |
|-----------|-----------|
| Redis 장애 | Redis Cluster 구성 (Master 3 + Replica 3), Sentinel 모드 |
| Kafka 장애 | Replication Factor 3, ISR 기반 내구성 보장 |
| RabbitMQ 장애 | Mirrored Queue + Dead Letter Queue (DLQ) 설정 |
| Payment 서비스 장애 | DLQ에 보관 후 복구 시 재처리, 임시 예약 TTL 만료 시 자동 취소 |
| DB 장애 | Read Replica 분리, 커넥션 풀 관리 (HikariCP) |

## 9. 모니터링 지표

- **처리량**: 초당 예매 요청 수 (TPS)
- **대기열**: 현재 대기 인원, 평균 대기 시간
- **Redis**: 메모리 사용률, 명령어 지연 시간
- **Kafka**: Consumer Lag (처리 지연 메시지 수)
- **RabbitMQ**: Queue 길이, 메시지 처리 속도
- **DB**: Query 응답 시간, 커넥션 풀 사용률
- **에러율**: 예매 실패율, 결제 실패율

## 10. 구현 범위 (Phase별)

### Phase 1 - 핵심 예매 기능
- [ ] User Service (회원가입/로그인/JWT)
- [ ] Event Service (공연 CRUD, Redis 캐싱)
- [ ] Ticket Service (재고 관리, 분산 락, 임시 예약)
- [ ] Redis 대기열 구현
- [ ] MySQL 스키마 설계 및 연동

### Phase 2 - 결제 및 알림
- [ ] Payment Service (RabbitMQ consumer, PG 연동)
- [ ] Notification Service (이메일/SMS)
- [ ] Kafka 이벤트 파이프라인 구성
- [ ] 결제 실패 시 재고 롤백 처리

### Phase 3 - 안정성 강화
- [ ] API Gateway Rate Limiting
- [ ] Redis Cluster 구성
- [ ] Kafka Replication 설정
- [ ] RabbitMQ DLQ 설정
- [ ] Circuit Breaker (Resilience4j)

### Phase 4 - 운영 및 모니터링
- [ ] Docker Compose 전체 환경 구성
- [ ] Prometheus + Grafana 대시보드
- [ ] 부하 테스트 (k6 또는 JMeter)
- [ ] 장애 시나리오 테스트
